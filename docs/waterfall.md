# Звіт: розробка проєкту за методологією Waterfall

## Мета роботи

Вивчити та зрозуміти основні принципи та етапи методології розробки Waterfall. Оволодіти ключовими концепціями цього підходу, включаючи послідовність етапів, взаємодію між ними та роль кожного етапу в життєвому циклі розробки програмного забезпечення. Отримати навички використання методології Waterfall для ефективного управління проєктом та розробки ІТ-продукту в умовах фіксованих вимог і чітко визначених етапів виконання робіт.

---

## Введення

**Waterfall (каскад)** — традиційна методологія розробки програмного забезпечення, яка передбачає послідовний перехід від одного етапу до іншого без повернення до попереднього етапу після його формального завершення. Етапи типово включають: оцінку та планування (вимоги), проектування, реалізацію, тестування та впровадження.

У рамках цієї роботи розробляється навчальний проєкт: **десктопний застосунок (Tauri + React)** для формування реліз-повідомлень у Discord на основі Jira-тікетів. Користувач вводить список тікетів (ключі або placeholder-посилання), отримує по них дані (у поточній версії — лише мок-дані) та генерує готовий текст повідомлення для публікації в Discord. Реальні Jira та Discord API не використовуються; достатньо мок-даних для демонстрації етапів Waterfall.

Нижче описано мінімум три етапи з детальним аналізом: **планування та вимоги**, **розробка**, **тестування**. Для кожного етапу підготовлена відповідна документація.

## 1. Планування та вимоги

### 1.1 Збір і фіксація вимог

На цьому етапі визначено мету продукту, цільових користувачів та перелік функціональних і нефункціональних вимог.

- **Мета продукту:** застосунок для формування реліз-повідомлень у Discord на основі Jira-тікетів (з мок-даними).
- **Користувачі:** розробники, реліз-менеджери, учасники команд, що використовують Jira та Discord.
- **Функціональні вимоги:** введення списку Jira-тікетів (ключі або placeholder); отримання даних по тікетах (мок); генерація тексту реліз-повідомлення для Discord; копіювання або відображення повідомлення.
- **Нефункціональні вимоги:** стек Tauri (Rust) + React (Tailwind, ShadCN UI); підтримка мови інтерфейсу (EN/UK) у налаштуваннях; десктопна платформа.

Деталізація зафіксована в документі **PRD** ([docs/prd.md](prd.md)).

### 1.2 Визначення обсягу, ресурсів та термінів

Визначено обсяг робіт, послідовність етапів та орієнтовні терміни виконання, а також ресурси та умовний бюджет.

- **Обсяг (scope):** UI для введення тікетів, логіка «fetch» мок-даних, генерація тексту повідомлення, копіювання/відображення результату, налаштування мови інтерфейсу.
- **Графік (schedule):** етапи Waterfall з орієнтовними термінами (планування та вимоги → проектування → реалізація → тестування → впровадження); загальна тривалість умовно близько 4,5 тижня.
- **Ресурси та бюджет:** один виконавець, умовні одиниці часу/бюджету без реальних фінансових витрат (навчальний мок).

Деталі зафіксовані в документі **Project Baseline** ([docs/project-baseline.md](project-baseline.md)).

### 1.3 Артефакти етапу «Планування та вимоги»

| Артефакт | Призначення |
|----------|-------------|
| [docs/prd.md](prd.md) | Документ вимог до продукту (PRD): мета, користувачі, функціональні та нефункціональні вимоги, обмеження. |
| [docs/project-baseline.md](project-baseline.md) | Базовий план проєкту: обсяг робіт та вимоги, графік виконання, ресурси та бюджет. |

Етап «Планування та вимоги» вважається завершеним після узгодження PRD та Project Baseline і переходу до етапу проектування/реалізації.

## 2. Розробка

### 2.1 Архітектура

Застосунок побудовано за схемою **десктопного клієнта** з двошаровою архітектурою:

- **Backend (Rust, Tauri 2):** виконується в нативному процесі, надає команду `fetch_tickets`, яка приймає список ключів тікетів і повертає мок-дані (структури `JiraTicket`: key, summary, status). Реальні зовнішні API (Jira/Discord) не використовуються.
- **Frontend (React + Vite):** односторінковий інтерфейс у WebView. Викликає Tauri command через `invoke("fetch_tickets", { keys })`, відображає список тікетів, генерує текст реліз-повідомлення та надає копіювання в буфер обміну. Налаштування (мова інтерфейсу) зберігаються в `localStorage`.

Потік даних: введення ключів → парсинг (кома/пробіл) → виклик `fetch_tickets` → відображення тікетів → генерація тексту за шаблоном → кнопка «Копіювати».

### 2.2 Технології та інструменти

| Шар / задача | Технологія |
|--------------|------------|
| Оболонка застосунку | Tauri 2 (Rust) |
| Frontend-фреймворк | React 19 + TypeScript |
| Збірка frontend | Vite 7 |
| Стилізація | Tailwind CSS 3 |
| Компоненти UI | ShadCN-стиль (CVA, clsx, tailwind-merge), кнопки та поля введення |
| Інтернаціоналізація | Власний модуль `i18n` (EN/UK), React Context для локалі, збереження в localStorage |
| Мок-дані | Фіксований набір структур у Rust, команда `fetch_tickets` |

### 2.3 Реалізований функціонал

- **Введення тікетів:** поле введення з підтримкою списку ключів через кому або пробіл (наприклад, PROJ-1, PROJ-2).
- **Отримання даних:** кнопка «Fetch» викликає команду Tauri `fetch_tickets`; для відомих ключів повертаються мок-записи, для невідомих — запис із позначкою [Mock].
- **Генерація реліз-повідомлення:** шаблон «Release v0.1.0» + список тікетів у вигляді «**KEY** — summary»; текст відображається в блоці під списком тікетів.
- **Копіювання:** кнопка «Copy» копіює згенерований текст у буфер обміну через `navigator.clipboard.writeText`.
- **Налаштування мови:** кнопка «Preferences» відкриває діалог з перемикачем мови (English / Українська); вибір зберігається в localStorage і застосовується до всіх підписів інтерфейсу.

### 2.4 Артефакти етапу «Розробка»

| Артефакт | Опис |
|----------|------|
| Репозиторій (гілка phase/development) | Вихідний код: `src-tauri/` (Rust, команди, мок-дані), `src/` (React-компоненти, i18n, UI). |
| Збірка | `npm run build` — frontend; `cargo build --manifest-path src-tauri/Cargo.toml` — backend; `npm run tauri build` — десктопний застосунок. |
| Документація | PRD та Project Baseline без змін; оновлено цей розділ у [docs/waterfall.md](waterfall.md). |

Етап «Розробка» завершено після реалізації переліченого функціоналу та оновлення водоного документу.

## 3. Тестування

### 3.1 Підхід до тестування

Для мок-версії застосунку проведено ручне тестування за попередньо підготовленими тест-кейсами. Автоматизовані e2e- або unit-тести в рамках навчального проєкту не впроваджувалися; акцент зроблено на покритті основних сценаріїв використання та фіксації результатів у документації.

### 3.2 Тест-кейси та сценарії

Детальний опис тест-кейсів наведено в документі [docs/testing.md](testing.md). Короткий перелік:

| ID | Об’єкт перевірки | Зміст |
|----|-------------------|--------|
| TC-1 | Введення тікетів | Прийняття одного/кількох ключів (кома, пробіл), порожній запит → повний мок-набір. |
| TC-2 | Отримання даних (мок fetch) | Відповіді для PROJ-1, PROJ-2, PROJ-3; fallback для невідомих ключів (наприклад, OTHER-99). |
| TC-3 | Генерація реліз-повідомлення | Поява блоку з текстом після fetch, коректний шаблон (заголовок, список тікетів), оновлення при зміні списку. |
| TC-4 | Копіювання в буфер | Кнопка «Copy» копіює текст; вставка в зовнішній редактор або Discord дає очікуваний результат. |
| TC-5 | Перемикання мови (EN/UK) | Діалог Preferences, зміна підписів інтерфейсу, збереження вибору в localStorage після перезапуску. |

### 3.3 Результати тестування

Перевірка виконана вручну на десктопній збірці (`npm run tauri dev`). Усі описані тест-кейси пройдено успішно: введення тікетів, виклик команди `fetch_tickets`, відображення мок-даних, генерація реліз-повідомлення та копіювання працюють відповідно до специфікації; перемикання мови та збереження налаштувань підтверджені. Критичних дефектів для мок-версії не виявлено. Документація тестування збережена в [docs/testing.md](testing.md) для подальшого використання при розширенні функціоналу або підключенні реальних API.

## 4. Висновок

Під час виконання лабораторної роботи проєкт було реалізовано за методологією Waterfall з чітким поділом на етапи: планування та вимоги → розробка → тестування. Нижче наведено короткий аналіз сильних та слабких сторін підходу з урахуванням отриманого досвіду та можливі напрямки покращення.

### 4.1 Сильні сторони Waterfall

- **Чітка структура та передбачуваність.** Фіксація вимог (PRD) і базового плану (Project Baseline) на початку дала зрозумілий обсяг робіт і послідовність кроків. Умови навчального проєкту (фіксована ідея, мок-дані) добре відповідали такому підходу.
- **Зручність документування.** Кожен етап має власні артефакти (PRD, Project Baseline, код, тест-кейси, звіт). Це спрощує звітність і передачу знань, зокрема для звіту з лабораторної роботи.
- **Простота управління для невеликого проєкту.** При одному виконавцеві та стабільних вимогах послідовне виконання етапів без повернень не створювало організаційних складнощів.
- **Чітке завершення етапів.** Формальний перехід від планування до розробки і далі до тестування допомагав тримати дисципліну і не змішувати фази (наприклад, не додавати нові вимоги під час кодування без оновлення документів).

### 4.2 Слабкі сторони та ризики

- **Низька гнучкість при зміні вимог.** Якби в ході розробки знадобилося підключити реальний Jira API або змінити формат реліз-повідомлення, класичний Waterfall вимагав би оновлення PRD і, можливо, повернення до етапу планування, що не завжди зручно в умовах обмеженого часу.
- **Відстрочена перевірка продукту.** Користувач (або викладач) бачить повноцінний результат лише після завершення етапу розробки та тестування. Ранній зворотний зв’язок щодо UX або змісту повідомлень обмежений.
- **Ризик надмірної документації.** Для невеликого навчального проєкту обсяг документів (PRD, Project Baseline, Waterfall-звіт, тест-документація) може здаватися надто великим порівняно з обсягом коду; у комерційних проєктах це може призводити до бюрократизації.

### 4.3 Ідеї для покращення роботи підходу

- **Гібридизація з елементами ітеративності:** зафіксувати базові вимоги та графік (як у Waterfall), але всередині етапу «Розробка» виконувати короткі цикли «реалізація → перевірка» (наприклад, по одній функції), щоб раніше виявляти помилки та коригувати UX.
- **Легкі огляди на межі етапів:** перед формальним переходом до наступної фази проводити короткий огляд артефактів (наприклад, перевірка PRD і Baseline перед стартом розробки), щоб зменшити ризик розбіжностей між документацією та реалізацією.
- **Чітке розмежування «зафіксовано» та «можливо зміниться»:** у PRD або Baseline явно позначити, які вимоги є стабільними, а які можуть уточнюватися під час розробки; це дозволить обмежено застосовувати зміни без повного відходу від Waterfall.

Загалом методологія Waterfall виявилася доречною для навчального проєкту з фіксованим набором вимог і послідовною реалізацією. Для проєктів із частою зміною вимог або потребою в швидкому зворотному зв’язку від користувачів доцільно розглядати гнучкі підходи (наприклад, Agile/Scrum) або поєднання чіткого планування з короткими ітераціями.
