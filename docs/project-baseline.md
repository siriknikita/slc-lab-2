# Project Baseline (Базовий план проєкту)

## 1. Обсяг робіт та вимоги (Scope and Requirements)

### 1.1 Короткий опис ідеї

ІТ-продукт — десктопний застосунок (Tauri + React), який дозволяє вводити список Jira-тікетів (ключі або placeholder-посилання), отримувати по них зведені дані (у навчальній версії — мок-дані) та генерувати текст реліз-повідомлення для публікації в Discord. Мета — спростити підготовку реліз-нотаток для команд, що використовують Jira та Discord.

### 1.2 Основні елементи розробки

- **UI для введення тікетів:** екран/форма з полем або списком для введення ідентифікаторів Jira-тікетів (наприклад, PROJ-1, PROJ-2).
- **Отримання даних по тікетах (мок):** логіка виклику «fetch» (через Tauri command або аналог), що повертає фіксовані мок-дані по тікетах без звернення до реального Jira API.
- **Генерація тексту реліз-повідомлення:** формування тексту за шаблоном (заголовок версії, список тікетів із summary тощо) на основі отриманих даних.
- **Копіювання / відображення повідомлення:** можливість скопіювати згенерований текст або переглянути його для ручного копіювання в Discord.
- **Налаштування мови інтерфейсу:** екран або модалка налаштувань (Preferences) з перемикачем мови інтерфейсу (EN/UK) та збереженням вибору.

Деталі функціональних та нефункціональних вимог описані в документі [PRD](prd.md).

## 2. Графік виконання (Schedule)

Етапи відповідають методології Waterfall. Орієнтовні терміни задані в умовних одиницях (тижні/дні) для навчального мока.

| Етап | Опис | Орієнтовна тривалість |
|------|------|------------------------|
| 1. Планування та вимоги | Збір і фіксація вимог, PRD, Project Baseline | 1 тиждень |
| 2. Проектування | Визначення архітектури (Tauri + React), екрани, потік даних | 0,5 тижня |
| 3. Реалізація | Scaffolding, мок-дані, UI введення тікетів, генерація повідомлення, налаштування мови | 2 тижні |
| 4. Тестування | Перевірка сценаріїв (введення, «fetch», генерація, копіювання, мова), фіксація результатів | 0,5 тижня |
| 5. Впровадження | Підготовка збірки для десктопу, документація для користувача (за потреби) | 0,5 тижня |

**Загальна орієнтовна тривалість:** близько 4,5 тижня (умовно, для навчального проєкту).

Залежності: кожен наступний етап розпочинається після завершення попереднього; повернення на попередній етап у класичному Waterfall не передбачається.

## 3. Ресурси та бюджет (Budget)

### 3.1 Людські ресурси

- **Один виконавець:** виконує ролі аналітика вимог, розробника (frontend + Tauri/Rust), тестувальника в рамках навчального проєкту.
- Розподіл часу орієнтовно відповідає графіку вище (планування, проектування, реалізація, тестування, впровадження).

### 3.2 Технічні ресурси

- Середовище розробки: ПК з підтримкою Tauri (Rust, Node.js, npm/pnpm).
- Інструменти: редактор коду, Git, збірка Tauri для цільової ОС.

### 3.3 Бюджет (умовний)

- У навчальному контексті реальні фінансові витрати не плануються.
- Бюджет виражено в умовних одиницях часу виконавця відповідно до графіка (орієнтовно 4,5 умовних тижні).
- Резерв на неочікувані зміни в умовах навчального завдання не формалізований; при необхідності коригування термінів оновлюється графік.

### 3.4 Контроль

- Відповідність обсягу робіт — за переліком елементів у розділі 1 та за PRD.
- Дотримання графіка — за етапами Waterfall та оновленням документації (зокрема `docs/waterfall.md`) по завершенні відповідних фаз.
