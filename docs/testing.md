# Документація тестування (Phase 3)

Документ описує ручні тест-кейси для перевірки мок-версії застосунку Jira Discord Release. Реальні Jira/Discord API не використовуються; усі перевірки базуються на мок-даних та локальному інтерфейсі.

---

## Передумови

- Застосунок зібрано та запущено: `npm run tauri dev`.
- Тестування виконується вручну на десктопі (Windows / macOS / Linux).

---

## Тест-кейси

### TC-1: Введення тікетів

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | Відкрити застосунок | Відображається заголовок «Jira Discord Release», поле введення та кнопка «Fetch» (або «Отримати» за української мови). |
| 2 | Ввести в поле один ключ: `PROJ-1` | Текст з’являється в полі введення. |
| 3 | Ввести кілька ключів через кому: `PROJ-1, PROJ-2, PROJ-3` | У полі відображається введений рядок. |
| 4 | Ввести ключі через пробіл: `PROJ-1 PROJ-2` | Рядок приймається (парсинг за комою та пробілом). |
| 5 | Залишити поле порожнім і натиснути «Fetch» | За планом: повертаються усі мок-тікети (PROJ-1, PROJ-2, PROJ-3). |

**Результат:** Поле введення коректно приймає один або кілька ключів, розділених комою або пробілом. Порожній запит повертає повний набір мок-даних.

---

### TC-2: Отримання даних (мок fetch)

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | Ввести `PROJ-1` і натиснути «Fetch» | Під формою з’являється список з одним тікетом: ключ, summary, status (наприклад, «Add login screen», «Done»). |
| 2 | Ввести `PROJ-1, PROJ-2` і натиснути «Fetch» | Відображаються два тікети з мок-даними. |
| 3 | Ввести невідомий ключ: `OTHER-99` | З’являється один тікет із summary типу «[Mock] Ticket OTHER-99» та статусом «Unknown». |
| 4 | Ввести суміш: `PROJ-1, OTHER-99` | Два тікети: один з фіксованого моку (PROJ-1), другий — згенерований для unknown ключа. |

**Результат:** Команда Tauri `fetch_tickets` коректно повертає мок-дані для відомих ключів (PROJ-1, PROJ-2, PROJ-3) та fallback-запис для невідомих ключів.

---

### TC-3: Генерація реліз-повідомлення

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | Після успішного «Fetch» з хоча б одним тікетом | Під списком тікетів з’являється блок «Release message for Discord» (або «Реліз-повідомлення для Discord»). |
| 2 | Переглянути текст повідомлення | Текст містить заголовок типу «## Release v0.1.0», підзаголовок «Included in this release» та список рядків «**KEY** — summary» для кожного тікета. |
| 3 | Змінити набір тікетів і знову натиснути «Fetch» | Текст реліз-повідомлення оновлюється відповідно до нового списку тікетів. |

**Результат:** Реліз-повідомлення генерується за шаблоном і відображається після отримання даних; оновлення списку тікетів оновлює текст.

---

### TC-4: Копіювання в буфер обміну

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | Після появи блоку з реліз-повідомленням натиснути кнопку «Copy» (або «Копіювати») | Кнопка коротко показує «Copied!» / «Скопійовано!». |
| 2 | Вставити вміст із буфера обміну (Ctrl+V / Cmd+V) у текстовий редактор або Discord | Вставлений текст збігається з відображеним реліз-повідомленням (заголовок, список тікетів). |

**Результат:** Текст реліз-повідомлення коректно копіюється в буфер обміну та готовий для вставки в Discord або інший канал.

---

### TC-5: Перемикання мови інтерфейсу (EN/UK)

| Крок | Дія | Очікуваний результат |
|------|-----|----------------------|
| 1 | Натиснути кнопку «Preferences» (або «Налаштування») | Відкривається діалог з перемикачем мови (English / Українська). |
| 2 | Обрати «Українська» і закрити діалог | Підписи в інтерфейсі змінюються на українські (наприклад, «Отримати», «Реліз-повідомлення для Discord», «Копіювати», «Налаштування»). |
| 3 | Перезапустити застосунок | Після перезапуску обрана мова (UK) зберігається (зчитана з localStorage). |
| 4 | Відкрити Preferences і обрати «English» | Підписи повертаються до англійської. |

**Результат:** Вибір мови в налаштуваннях змінює підписи інтерфейсу; вибір зберігається між сесіями.

---

## Підсумок

Усі описані сценарії покривають основний функціонал мок-версії: введення тікетів, отримання мок-даних, генерацію реліз-повідомлення, копіювання та перемикання мови. Тестування виконано вручну; автоматизовані e2e-тести у рамках навчального проєкту не передбачені.
